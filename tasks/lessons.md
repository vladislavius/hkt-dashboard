# Lessons — hkt-dashboard

## 2026-04-22 — Перед деструктивом предлагать бекап

**Контекст:** План миграции данных с Supabase в JSON. Проверил 1:1 совпадение и сразу предложил `TRUNCATE` без шага «сделать бекап». Юзер спросил *«бекап супабейз делаем? или не надо?»* — значит шаг не был озвучен.

**Правило:** В любом плане, где есть `TRUNCATE`/`DELETE`/`DROP` или перезапись файлов, **по умолчанию** включать пункт «сделать бекап» — даже если аргумент «данные есть в другом месте» выглядит убедительно. Пользователь должен видеть опцию, а не спрашивать о ней.

**Как применять:** Перед озвучиванием деструктивного плана — вписать в список шагов «0. Дамп/снапшот в файл» или явно сказать «бекап не нужен потому что X» с обоснованием. Не оставлять шаг невидимым.

## 2026-04-22 — Git fetch перед обвинениями CI

**Контекст:** Юзер пожаловался что DAD/PQC не отображаются. Я посмотрел `git log` и увидел что последний DAD commit в 15:56, 8 часов назад → сделал громкий диагноз «GH Actions квота исчерпана». На деле при попытке push я получил конфликт с свежим bot-коммитом `bc2306b auto: update DAD flight data 2026-04-22 17:42 ICT`, которого не было в моём `git log` потому что я его не зафетчил.

**Правило:** Перед тем как заявить что CI/cron/сервис сломан или квота кончилась — `git fetch origin` и `git log origin/main` обязательно. Bot-коммиты случаются параллельно с работой, и local `git log` становится устаревшим за минуты.

**Как применять:** Любая диагностика «workflow не отработал» / «данные устарели» / «сервис не пингуется» начинается с **fetch + origin лог**. Только после этого — гипотезы о квоте, сбоях и откатах.

## 2026-04-22 — Когда юзер предлагает откат — явно сказать «не поможет, потому что»

**Контекст:** Юзер: *«если база в супабейз не влияла на размер может вернуть все на место? ты же бекап делал»*. Я мог бы ответить «ок, откатим» и потратить 10 минут на восстановление из бекапов — бесполезно, потому что фронт Supabase не читает.

**Правило:** Если юзер предлагает откат/переделку, и я **точно знаю что это не решит его проблему** — остановить его явным объяснением, не соглашаться на бесполезную работу. Даже если откат «безопасен» — он отнимает время от реальной диагностики.

**Как применять:** В ответ на «может вернуть?» — сначала описать текущие симптомы и как они соотносятся с предложенным откатом. Если откат не коррелирует с симптомом — сказать прямо «откат не поможет, потому что X, реальная причина — Y». Только если реальная причина неясна — обсуждать откат как one из гипотез.

## 2026-04-23 — «Раньше работало» — проверять git-историю в аналогичный момент прошлого дня

**Контекст:** Юзер заявил «раньше различал план vs выполнено». Я чуть не начал фиксить как регрессию. Проверил git-историю — `accumulated_2026-04-22.json` первый снапшот (00:08 ICT 22 Apr) имел идентичную картину сегодняшнему (count=74, landed=0, 65 scheduled). Это не регрессия, это всегда так было. Юзер просто не видел сайт в 00:xx ICT раньше — заходил днём когда landed уже рос.

**Правило:** Если юзер говорит «раньше было иначе» — не чинить на веру. Найти git-коммит аналогичного состояния в прошлом и сравнить. Для ежедневных snapshot'ов: `git log --oneline --all -- <file>` → первый коммит того же времени суток вчерашнего дня.

**Как применять:** Перед любым «багфиксом на жалобу о регрессии» — 5 минут на `git show <old-commit>:<path> | jq ...` и сравнение полей. Если одинаково — это не регрессия, это feature (или всегдашний баг). Озвучить юзеру с доказательствами, потом решать фиксить или нет.

## 2026-04-23 — Фильтровать данные в ОДНОЙ точке, а не в каждом виджете

**Контекст:** Чтобы «прилёты» показывали только реально прилетевшие, я добавил filter ТОЛЬКО в `mkMetricInline` (большая карточка). Donut, country breakdown и CIS ticker читали raw `d.arrivals.count` = 74. Юзер увидел «3» в карточке и «74» в donut → разозлился, матерился, сказал «делаем код ревью, фиксим корень».

**Правило:** Если нужно изменить как данные интерпретируются на странице — мутировать их **в ОДНОЙ точке at data-load boundary** (сразу после `fetch().then(r=>r.json())`), а не в render-функциях каждого виджета. Все downstream widgets автоматически увидят одинаковую картину. Single source of truth.

**Как применять:** Перед правкой render-кода — найти где данные загружаются/парсятся (обычно `fetch` + `data = await res.json()`) и сделать transform там. Пример: `data.today = rebuildTodayFromList(data.today)` после await. Тогда card/donut/list/breakdown/ticker — все читают уже корректные числа. Не дублировать filter в каждом рендере.

## 2026-04-23 — При массовых правках HTML/JS — Python-скрипт + regex вместо ручных Edit'ов

**Контекст:** Нужно было применить одну правку в 4 HTMLs. Раньше делал через 4 отдельных `Edit` tool calls — медленно, риск typo. Сейчас сделал через Python inline-скрипт с `re.sub` — вся правка за 1 Bash-вызов, с автоматической verification в конце (`grep -c "rebuildTodayFromList"`).

**Правило:** Если одна и та же правка требуется в 2+ похожих файлах — написать Python-инлайн через Bash, а не делать Edit × N. Включить verification (grep count, status check) в тот же скрипт.

**Как применять:** Python блок через `python3 <<'PY' ... PY` + regex для поиска точек вставки + `p.write_text(s)` для записи. Гарантирует идентичность правок и не ломает YAML/JSON от редких unicode edge cases.

## 2026-04-23 — Читать return value функции ДО диагноза

**Контекст:** Я несколько часов объяснял юзеру что «GTT возвращает только 24h окно, вечерние рейсы потеряны». На самом деле GTT возвращал 161 рейс включая вечерние 22 Apr (в 24h окне от момента запроса они ещё были), но `analyze_gtt` кладёт всё под один ключ `{date_str: ...}` (= today), а collector делал `a_res.get(today)` → выбрасывал flights с arr_time из прошлого/будущего. Баг был НЕ в источнике, а в extraction-логике.

**Правило:** Когда диагностируешь «данные пропали» / «источник не отдаёт» — **прочитать полностью return shape функции которая парсит ответ**. Не верить промежуточным диагнозам («API режет retention»), пока не увидел: (1) raw count (161 rec received), (2) structure returned (dict keyed by date_str ONCE), (3) consumer logic (`a_res.get(today)` — берёт только один ключ). Часто бюджет теряется в consumer'е, а не в источнике.

**Как применять:** Trace chain `source → parser → consumer → output`. На каждом шаге проверить что данные не теряются. «GTT returned 164 but only 78 made it into file» — здесь между GTT и файлом что-то режет. Не предполагать что парсер идеален — прочитать его.
